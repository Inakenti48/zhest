## Project Summary
Система управления производством для жестяного цеха. Позволяет отслеживать складские остатки, продажи, заказы и посещаемость сотрудников. Включает в себя каталог изделий с возможностью онлайн-заказа.

## Tech Stack
- **Framework**: Next.js (App Router)
- **Runtime**: Bun / Node.js
- **Database**: PostgreSQL (pg) / SQLite (better-sqlite3) hybrid
- **Styling**: Tailwind CSS
- **Animations**: Framer Motion
- **Icons**: Lucide React

## Architecture
- `src/app`: Маршруты и страницы Next.js.
- `src/components`: UI компоненты (каталог, формы, обертки).
- `src/lib`: Логика базы данных (`db.ts`), действия сервера (`product-actions.ts`, `auth-actions.ts`).
- `public/uploads`: Локальное хранилище для загруженных изображений товаров.
- `postgre.sql`: Схема и данные для миграции на PostgreSQL.
- `docker-compose.yml`: Конфигурация для локального запуска PostgreSQL.

## User Preferences
- Использовать только локальные решения или собственный хостинг (PostgreSQL, локальное хранение файлов).
- **КАТЕГОРИЧЕСКИ ЗАПРЕЩЕНО** использовать Supabase или внешние облачные БД (Neon и т.д.).
- **НИКОГДА НЕ ЗАНОСИТЬ НИЧЕГО В SUPABASE** — даже если есть подключённый проект, НЕ использовать его. Только SQLite (zhest.db).
- Все файлы проекта должны храниться в папке `zhest`.
- **НЕ ДЕЛАТЬ НИЧЕГО ОТ СЕБЯ** — всегда спрашивать перед удалением фото, файлов или данных.
- **СПРАШИВАТЬ РАЗРЕШЕНИЕ** перед любым деструктивным действием (удаление, замена, перезапись).
- **ЕСЛИ СКАЗАЛ ЧИСЛО — ДЕЛАЙ СТОЛЬКО**: Когда пользователь говорит конкретное число (например "5 фото", "3 новых"), делать РОВНО столько. Это может быть: сохранить фото, добавить к уже сохранённым товарам, занести в БД, поправить данные. Число = команда.
- **НЕ БЫТЬ ДОЛГИМ**: Когда пользователь присылает фото для добавления к товару — сразу добавлять без лишних слов и объяснений. Минимум текста, максимум действий.
- **ЛОГИКА СМЕНЫ ФОТО**: При каждом занесении фото или товара (в том числе при замене) — картинка должна переключаться на соответствующую при выборе цвета. Это обязательное условие.

## Project Guidelines
- Никаких зависимостей от Supabase. Ни одной строки кода.
- База данных по умолчанию: `zhest.db` (SQLite).
- Переключение на PostgreSQL через `DATABASE_URL` в `.env`.
- Изображения товаров сохраняются в `public/uploads`.
- Основной язык интерфейса: Русский.

## Common Patterns
- Использование асинхронных Server Actions для работы с БД (поддержка pg).
- Анимации появления элементов с помощью Framer Motion.
- Динамическая загрузка категорий на основе имеющихся товаров.

## КРИТИЧНЫЕ ПРАВИЛА ДЛЯ ДОБАВЛЕНИЯ ТОВАРОВ (ЗАПОМНИТЬ!)

**При слове "заноси" в запросе — это команда добавить товар в БД!**

### Структура variants для КАЖДОГО товара:
```json
{
  "name": "RAL XXXX Название цвета",
  "color": "RAL XXXX Название цвета", 
  "colorCode": "#HEXКОД",
  "image_url": "/uploads/filename.jpeg"
}
```

### Обязательные поля:
1. **name** — полное название (RAL номер + название цвета)
2. **color** — то же самое для совместимости
3. **colorCode** — HEX код цвета для окрашивания кнопки выбора
4. **image_url** — путь к фото товара этого цвета

### Логика работы:
- При выборе цвета — картинка товара меняется на соответствующую
- Кнопка цвета окрашена в colorCode
- Если фото больше 10 — ЖДАТЬ пока пользователь загрузит ВСЕ фото
- Даже если фото придут во втором запросе — сначала ждать все фото

### Пример как у "Желоб квадратный":
```json
[
  {"name":"RAL 8017 Коричневый","color":"RAL 8017 Коричневый","colorCode":"#5c4a3d","image_url":"/uploads/gutter-brown.jpeg"},
  {"name":"RAL 3005 Вишневый","color":"RAL 3005 Вишневый","colorCode":"#5E2028","image_url":"/uploads/gutter-cherry.jpeg"}
]
```

## БЫСТРОЕ ДОБАВЛЕНИЕ ТОВАРОВ ЧЕРЕЗ КНОПКУ + В КАТАЛОГЕ

### Как пользоваться:
1. В каталоге рядом с категориями есть зелёная кнопка **+**
2. Нажимаешь — открывается модальное окно
3. Вводишь **название товара** (например: "Угол желоба внутренний")
4. Вводишь **категорию** (например: "Квад водосточка")
5. Загружаешь до **15 фото** за раз
6. RAL номер и цвет **определяются АВТОМАТИЧЕСКИ ПО ЦВЕТУ ФОТО** (анализ изображения через sharp)
7. Нажимаешь "Добавить товар" — все фото сохраняются в `public/uploads`, товар создаётся в БД с variants

### Автоматическое определение RAL по цвету фото:
- Система анализирует доминирующий цвет в центре изображения
- Сопоставляет с базой RAL цветов и находит ближайший
- Автоматически подставляет RAL номер, название цвета и HEX код
- Имя файла НЕ важно — RAL определяется по самому изображению

### ВАЖНО — Ничего не теряется:
- Фото сохраняются в `public/uploads` с timestamp именами
- Товар сохраняется в БД SQLite (zhest.db)
- После успешного добавления страница перезагружается
